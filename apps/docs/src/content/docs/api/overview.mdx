The PING Platform API is powered by [Convex](https://convex.dev), a serverless backend that provides real-time queries, transactional mutations, and side-effect actions. All API calls are made via HTTP POST requests using a JSON-RPC style interface.

## Function types

Convex exposes three types of functions:

| Type | Purpose | Endpoint | Characteristics |
|------|---------|----------|-----------------|
| **Query** | Read data | `POST /api/query` | Deterministic, cached, real-time subscriptions |
| **Mutation** | Write data | `POST /api/mutation` | Transactional, serialized |
| **Action** | Side effects | `POST /api/action` | Can call external services, not transactional |

## Base URL

Your Convex deployment URL serves as the base for all API calls. It is typically set as the `NEXT_PUBLIC_CONVEX_URL` environment variable:

```
https://your-deployment.convex.cloud
```

## Request format

All requests use `POST` with a JSON body containing three fields:

```json
{
  "path": "module:functionName",
  "args": { },
  "format": "json"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `path` | `string` | Function reference in the form `module:functionName` (e.g., `channels:list`) |
| `args` | `object` | Arguments to pass to the function |
| `format` | `string` | Always `"json"` |

### Example request

<Tabs>

```bash
curl -X POST https://your-deployment.convex.cloud/api/query \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <your-jwt-token>" \
  -d '{
    "path": "channels:list",
    "args": {},
    "format": "json"
  }'
```

## Response format

All responses return a JSON object with the following shape:

### Success response

```json
{
  "status": "success",
  "value": { }
}
```

### Error response

```json
{
  "status": "error",
  "errorMessage": "Human-readable error description",
  "errorData": { }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `status` | `"success"` or `"error"` | Whether the call succeeded |
| `value` | `any` | The return value of the function (only on success) |
| `errorMessage` | `string` | Error description (only on error) |
| `errorData` | `object` | Additional error context (only on error) |

## Common error codes

| Error message | Cause |
|---------------|-------|
| `"Not authenticated"` | Missing or invalid JWT token |
| `"User not found"` | Token is valid but no matching user record exists |
| `"Not a member of this channel"` | Attempting to access a channel without membership |
| `"Channel name already taken"` | Creating a channel with a duplicate name |

<Aside type="tip">
When using the Convex client SDK (e.g., `convex/react`), you do not need to construct HTTP requests manually. The SDK handles request formatting, authentication, and real-time subscriptions automatically.
</Aside>

## Rate limits

Convex enforces built-in rate limits at the platform level. For most workloads, these limits are generous and unlikely to be hit. If you encounter rate-limiting errors, consider batching requests or using Convex subscriptions for real-time data instead of polling.

## Pagination

Some endpoints (such as `messages:list` and `directMessages:list`) support cursor-based pagination. Pass a `paginationOpts` object:

```json
{
  "path": "messages:list",
  "args": {
    "channelId": "jd7abc123...",
    "paginationOpts": {
      "numItems": 25,
      "cursor": null
    }
  },
  "format": "json"
}
```

The response includes pagination metadata:

```json
{
  "status": "success",
  "value": {
    "page": [ ],
    "isDone": false,
    "continueCursor": "eyJ..."
  }
}
```

Pass `continueCursor` as the `cursor` in the next request to fetch the next page.
